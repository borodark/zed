# S6 plan: tarball deployment in the converge engine

**Status:** design draft, April 2026.
**Scope:** turn the existing `app:create` step's `:no_tarball` stub
into a real release-deployment pipeline. Jail-aware. Idempotent.
Health-gated. The piece that gets `MyInfra.Demo.converge/0` from
"jails created" to "five BEAMs running and clustered."
**Effort:** ~3 days, broken into five sub-steps that land
independently. Critical path = sub-step #2 (executor wiring) — the
others are additive.

---

## What "deploy a tarball" means here

For each BEAM jail in the demo:

```
HOST                                          JAIL
────                                          ────
~/zed/demo-releases/zedweb/zedweb-0.1.0.tar.gz       (the artifact)
                            │
              cp ──────────►│
                            │
              <bastille_jails_dir>/zedweb/root/srv/zedweb/releases/0.1.0/
                            │ (extract here)
              symlink ──────►/srv/zedweb/current → releases/0.1.0
                            │
              service ──────► /srv/zedweb/current/bin/zedweb daemon
```

- **Target path inside the jail:** `/srv/<app>/releases/<vsn>/`
- **Symlink:** `/srv/<app>/current` → `releases/<vsn>/`
- **Service entry:** rc.d script generated by zed reads `current/bin/<app>`
- **Cookie file:** nullfs-mounted from host's `<base>/zed/secrets/`
- **Cluster artifact:** nullfs-mounted from host's `<base>/zed/cluster/`
- **Older versions:** retained per `snapshots keep N` for rollback

Single-host shortcut: jails are local filesystems on the host, so
"copy into the jail" is `cp -r` to the bastille jail root path. No
network involved. Future multi-host iteration replaces this with
`zfs send | ssh zfs receive`.

## Decisions to lock

| # | Decision | Default lean |
|---|---|---|
| 1 | Where tarballs live on the host | `~/zed/demo-releases/<app>/<app>-<vsn>.tar.gz` (operator-curated; manifest doc lists the canonical layout) |
| 2 | Tarball format | The plain `mix release` directory (NOT tar'd) — simpler, handles fine on local fs, skips compress/decompress overhead. DSL field renamed `release_path` (already exists) to a release-DIRECTORY path. Future iteration adds an actual tarball form for cross-host. |
| 3 | Extract method | `cp -r` on the host (jail root is a host path). Avoids `bastille cmd` for the extract — faster + no jail-internal tar dependency. |
| 4 | Service start mechanism | rc.d script template per-app, `bastille service <jail> <svc> start`. The script execs `/srv/<app>/current/bin/<app> daemon` as the app's run-as user. |
| 5 | Run-as user inside the jail | Default `zed` (single user per jail; created by the host-bring-up of the jail). DSL field `app_user` lets the operator override per-app. |
| 6 | Cookie + cluster artifact delivery | nullfs ro mount from host's `<base>/zed/secrets/` and `<base>/zed/cluster/` into the jail's `/var/db/zed/`. Mac1's `nullfs_mount` DSL verb already exists. |
| 7 | Health gating | After service start, poll `:health` entries with a deadline. Demo accepts `:http` (GET, expect status) and `:beam_ping` (Node.connect + `:rpc.call(:erlang, :node, [])`). Other types deferred. |
| 8 | Rollback | `zfs rollback <data_dataset>@pre-deploy-<vsn>` + flip `current` symlink back. Snapshot taken before the new release lands. |
| 9 | Idempotency | Re-running `converge` with the same version is a no-op (current symlink already points there + property `com.zed:version` matches). New version triggers full pipeline. |

## Sub-iterations

### S6.1 — DSL + IR fields for release deployment

**~0.5d.** Light extension to the existing `app` block.

**New DSL fields** (validator additions, no new verb):

```elixir
app :zedweb do
  release_path "/home/io/zed/_build/prod/rel/zedweb"   # host path
  version "0.1.0"
  app_user "zed"                                       # optional, default = app name
  cookie {:secret, :demo_cluster_cookie, :value}
  health :http, url: "http://10.17.89.10:4040/health", expect: 200
end
```

**Validator additions:**
- `release_path` — must be a binary, must point at an existing directory at compile time (or at converge time? — leans converge; allows `mix compile` on a different host than where the release lives).
- `app_user` — must be a binary or atom; default `to_string(app_id)`.
- Inline-app inside jail (Mac1's S3) already passes these through.

**Files touched:** `lib/zed/ir/validate.ex` (additive checks),
`lib/zed/dsl.ex` (no change — verbs already accepted as kv).

### S6.2 — Plan + Executor for `app:deploy`

**~0.5d.** Replace the `{:ok, :no_tarball}` stub.

**Plan step `:app` action `:create` carries new args:**
- `release_path` — host path to the release directory
- `jail_name` — which jail this app deploys into (derived from `contains` reverse-lookup, or from inline-app's parent jail)
- `app_user`
- `version`
- `bastille_jails_dir` — config (default `/usr/local/bastille/jails`)

**Executor flow:**
1. Compute `target = <bastille_jails_dir>/<jail>/root/srv/<app>/releases/<vsn>`
2. If `target` exists AND `current` symlink already points there → `{:ok, :no_op}` (idempotent).
3. Otherwise: snapshot (already handled by Plan ordering),
   `File.mkdir_p!(target)`, `File.cp_r!(release_path, target)`,
   atomic-flip the `current` symlink (rm+symlink in one shot).
4. Stamp ZFS properties (already handled by `stamp_app_properties`).

**Files touched:** `lib/zed/converge/plan.ex` (extend
`build_app_deploy_step/3` args), `lib/zed/converge/executor.ex`
(real `deploy_release` implementation).

### S6.3 — rc.d service generation + `jail_svc` real wiring

**~0.5d.** Replace the `{:jail_svc_pending, ...}` stub.

**Per-app rc.d script** rendered to
`<bastille_jails_dir>/<jail>/root/usr/local/etc/rc.d/<app>`:

```sh
#!/bin/sh
# zed-managed — do not edit; regenerated by every converge.
. /etc/rc.subr

name="<app>"
rcvar=<app>_enable
command="/srv/<app>/current/bin/<app>"
command_args="daemon"
<app>_user="<app_user>"
<app>_chdir="/srv/<app>/current"
<app>_env="HOME=/srv/<app>/current LANG=en_US.UTF-8"

load_rc_config $name
: ${<app>_enable:="YES"}

run_rc_command "$1"
```

**Executor flow for `:jail_svc`:**
1. Render the rc.d script from a small EEx template.
2. Write to the jail's rc.d path (host-side cp).
3. `bastille service <jail> <app> enable` (sysrc inside the jail).
4. `bastille service <jail> <app> start`.
5. Return `{:ok, {:service_started, app, jail}}`.

**Files touched:** new `lib/zed/converge/rc_d.ex` for templating,
`lib/zed/converge/executor.ex` (`:jail_svc` clause).

### S6.4 — Health-check polling

**~0.5d.** New plan step `:health_wait` slotting after `:jail_svc`.

**Per-health-check step:**
- `:http` — `httpc:request/4` against the URL with timeout. Repeat
  every 1s up to 60s. Pass when status matches `expect:`.
- `:beam_ping` — `Node.connect(target_node)` + `:rpc.call(target,
  :erlang, :node, [])`. Pass when reply matches `target_node`.

**Plan integration:** insert `health_wait` steps after the
corresponding `jail_svc:start`. Sort priority slot:
`jail_svc → health_wait → service`.

**Failure semantics:** timeout exits the converge with
`{:error, {:health_timeout, app, kind, last_error}}`. Plan
caller decides whether to roll back; demo wires it to roll back
the failing app's symlink + snapshot.

**Files touched:** `lib/zed/converge/plan.ex` (new step builder),
`lib/zed/converge/executor.ex` (new clause), new
`lib/zed/beam/health.ex` for the polling primitives.

### S6.5 — Live `:bastille_live` end-to-end test

**~1d.** The integration moment.

**Test:** `test/zed/converge/release_deploy_live_test.exs`,
tagged `:bastille_live`. Uses a tiny pre-built BEAM release fixture
(`test/fixtures/echo_app/`) — a single-supervisor app that registers
`:global` named `:echo_app_node`.

**Flow:**
1. Create a unique jail name (`zed-deploy-test-<rand>`)
2. Build the demo IR with that jail + an `echo_app` app referencing
   the fixture's release path
3. Run `MyInfra.Demo.converge()` → expect `{:ok, _}`
4. Connect to the jail's BEAM via `Node.connect(:"echo_app@<ip>")`
5. Assert `:rpc.call(target, :global, :whereis_name, [:echo_app_node])`
   returns a non-nil pid
6. Tear down: `Bastille.destroy/2` (handles the post-condition
   check from A5.1)

**Files touched:** new test file + `test/fixtures/echo_app/`
release artifact (smallest-possible Mix release, ~50 LOC + a
build script that produces the tarball during test setup).

## What this DOES NOT cover

- **Multi-host tarball delivery.** Single-host only; future
  iteration uses `zfs send | ssh zfs receive` for the release
  dataset.
- **Tarball signing / fingerprint verification.** Operators trust
  what they put in `~/zed/demo-releases/`; future iteration adds
  a `:fingerprint` field on `release_path` checked at deploy time.
- **Hot code upgrade.** Each version is a fresh release; restart
  is a stop+start. Hot upgrades are a separate iteration after
  the rolling-restart pattern lands.
- **Per-app log rotation.** The rc.d script writes to default
  syslog; log layout is operator-managed for now.
- **GUI feedback during converge.** zedweb's dashboard doesn't yet
  stream converge progress; the demo just blocks on
  `MyInfra.Demo.converge()` and prints to stdout.

## Risks

1. **`bastille cmd <jail> ./bin/<app> daemon` doesn't always
   detach.** Mitigated by going through rc.d (S6.3) instead of
   `bastille cmd` for the start verb. rc.d's `run_rc_command`
   handles daemonisation correctly.
2. **Cookie / cluster artifact race.** The artifact has to exist
   in the jail's nullfs-mounted view BEFORE the BEAM boots. Plan
   sort already orders `cluster_config` and `jail_mount` before
   `jail_svc:start`; verify this on the live test.
3. **Per-jail Erlang runtime version.** All five demo apps target
   OTP 27, the jail pkg `erlang-runtime27` provides it. mac-247's
   S5a notes that Plausible required this; mac-248 confirmed
   livebook + exmc need it too. Validate at converge time:
   `bastille pkg <jail> info erlang-runtime27` returns success
   before app:deploy fires.
4. **libcluster Epmd needs `epmd` running in each jail.** Bastille
   jails don't auto-start `epmd`; OTP starts it on demand when the
   first distribution-aware BEAM boots. The first node in the
   cluster pays a small startup tax. No action needed; flag in
   the blog as "expected behaviour."
5. **`File.cp_r` on a 100MB release directory takes seconds.** The
   demo's five apps are small; total deploy time should be under
   a minute. Watch this number and surface it in the blog if
   higher than expected.

## Acceptance criteria

- [ ] `MyInfra.Demo.converge()` runs end-to-end on a fresh Mac Pro
      and exits `{:ok, _}` with no operator intervention beyond
      `pkg install` + `host-bring-up.sh`.
- [ ] After converge, `Node.list/0` from any of the five jails
      returns the other four.
- [ ] `bastille list` shows seven jails (5 BEAM + pg + ch), all
      `Up`.
- [ ] zedweb dashboard at `http://<host_ip>:443/` (after pf rdr)
      shows the cluster panel green for all five nodes.
- [ ] `mix run -e 'MyInfra.Demo.rollback("@pre-converge")'` returns
      the host to its pre-converge state in under 30 seconds.
- [ ] Re-running `converge` with no IR changes is a no-op (all
      steps return `{:ok, :no_op}`).
